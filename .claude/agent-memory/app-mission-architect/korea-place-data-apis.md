2026-07-14 검증. **"맛집/장소 추천" 기능을 스코핑할 때 반드시 먼저 확인할 것.** 사용자는 보통 "평점 높고 지금 열려 있는 곳"을 당연히 되는 줄 알고 요구하는데, **국내 API로는 둘 다 불가능하다.**

| 데이터 | 카카오 로컬 | 네이버 지역검색 | **Google Places (New)** |
|---|---|---|---|
| 이름·주소·좌표·카테고리·거리 | ✅ | ✅ | ✅ |
| **평점 / 리뷰 수** | ❌ | ❌ | ✅ `rating` / `userRatingCount` |
| **영업시간 / 지금 열림** | ❌ | ❌ | ✅ `regularOpeningHours` / `currentOpeningHours.openNow` |
| 가격대 | ❌ | ❌ | ✅ `priceLevel` |

- **카카오 로컬은 영업시간조차 안 준다.** 카카오맵 상세페이지(`place_url`)에만 있고 **그건 크롤링**이다 (카카오 데브톡에서 반복 확인된 사실).
- **Google Places는 한국에서 정상 동작한다.** 한국에서 막힌 건 **길찾기(Directions/도보·자동차 안내)** 이지 **장소 정보가 아니다.** 국가 커버리지 제한 목록에 한국 없음.
  - 단, `rating`/`userRatingCount`/`openingHours` 는 **Enterprise SKU** → 신용카드 등록 필요(무료 한도 내 청구 0원).
  - 약점: **국내 로컬 맛집은 구글 리뷰가 얕다.** 리뷰 20건 미만이면 평점을 신뢰하지 말 것.

**권장 조합:** 한국 주소·POI → 좌표는 **카카오**(정확도 우위), 평점·영업시간은 **Google Places**. 용도별로 나눠 쓴다.

## 스코핑 원칙 — 목적지를 실제로 '실행'하는 앱에서 LLM 환각

차량 내비·예약처럼 **결과가 물리적으로 실행되는** 앱에서는 **LLM이 장소를 생성하게 두면 안 된다.** 폐업했거나 존재하지 않는 가게가 실제로 목적지에 꽂힌다.
→ **후보는 반드시 API 응답으로만 제한(grounding)하고, LLM은 그 안에서 선별·설명만 하게 한다.**
