**규칙:** 새 앱 아이디어를 스코핑하기 전에, 저장소에서 그 앱이 **이미 존재하는지 먼저 검색한다.** 사용자가 언급한 키(예: `RAPA\.env`)나 도메인 키워드로 `Grep`/`Glob` 을 돌려 기존 구현을 확인한 뒤 질문을 설계한다.

**Why:** 2026-07-13, 사용자가 "교육 수료증·수강확인증·모범상을 자동 발급하고 DB에 번호를 누적하는 앱을 만들고 싶다"고 요청했는데, `RAPA/` 에 **증서 5종 + HRD-Net 명단 자동조회 + 시트 적재 + 번호 자동채번**을 모두 갖춘 앱이 이미 배포까지 되어 있었다. 이걸 모르고 질문했으면 "무엇을 만들까"만 물었을 것이고, 결과물은 이미 있는 앱의 복제였을 것이다. 실제로 물어야 할 것은 **"이미 있는데 왜 새로 만드는가 = 무엇이 부족한가"** 였다.

**추가 교훈:** 사용자가 외부 API 능력에 대해 말하는 내용이 내 메모리와 **충돌하면, 메모리가 아니라 사용자를 의심하지 말고 라이브로 검증하라.** 이 건에서 메모리("HRD-Net 은 훈련생 PII 안 줌")가 틀렸고 사용자가 맞았다. 30초짜리 API 호출로 판가름났다.

**How to apply:** 1라운드 질문을 보내기 전에 저장소를 훑는다. 기존 구현을 찾으면 **1라운드 질문 자체를 "기존 앱의 어떤 점이 불만인가"로 바꿔라.** 그러면 미션이 "복제"가 아니라 "개선 delta" 로 잡힌다.
