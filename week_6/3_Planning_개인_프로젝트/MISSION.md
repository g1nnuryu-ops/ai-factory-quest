## 미션 한 문장

> **훈련생 명단 확인부터 증명서 작성·채번·발급대장 기입까지 전부 수기로 하던 증명서 발급을, 고용24(HRD-Net) 기관 API에 직접 연결해 「과정 선택 → 교육생 선택 → 발급」 한 흐름으로 끝내고, 그 결과가 구글시트 발급대장에 자동으로 한 줄 쌓이게 한다.**

## 한 문단 요약 — 문제 · 타겟 · 해결

**문제.** 증명서 한 장을 내보내려면 **모든 단계가 수기였다.** 훈련생 이름·생년월일을 명단에서 찾아 옮겨 적고 → 양식에 타이핑하고 → 발급대장을 열어 마지막 번호를 눈으로 확인해 다음 번호를 붙이고 → 출력한 뒤 → 대장에 한 줄을 손으로 기입한다. **옮겨 적는 자리마다 오타가 나고, 눈으로 매기는 번호는 겹친다.** 그런데 한 번 나간 공식 문서는 회수할 수 없다.

**타겟.** 이 앱을 쓰는 사람은 딱 한 명, **나 자신이다** — *수강확인증 요청이 올 때마다, 그리고 과정이 끝날 때마다 같은 옮겨적기를 반복하고 있는 RAPA 교육사업 담당자* (증서 명의는 센터 구성원 14명 중에서 고른다). 40명 과정의 39번째쯤 되면 손이 기계적으로 움직이고 **바로 그때 이름이 밀리거나 번호가 겹친다.**

**해결.** 훈련생 명단은 **고용24(HRD-Net) 기관전용 API가 이미 갖고 있다.** 과정만 고르면 이름·생년월일·출결·수료상태가 그대로 내려온다. 여기에 **서버 채번**과 **구글시트 발급대장 자동 기입**을 붙여, **증명서 출력과 대장 기록이 버튼 한 번에 동시에 끝난다. 사람이 옮겨 적는 단계가 아예 사라진다.** 훈련생의 이름·생년월일은 이 흐름 안에서만 흐르고 별도 명단 파일로 복제되지 않는다.

> 📌 **왜 사서 쓰지 않는가 — 핵심 기능을 SaaS가 대체할 수 없기 때문이다.** 경쟁 서비스 4곳(칼리지스·브루프·Accredible·모두싸인)을 직접 조사한 결과, **4사 모두 "명단은 기관이 CSV로 올려라"가 대전제**다. 즉 **내가 없애려는 바로 그 수기 단계(명단 옮겨적기)를 SaaS는 오히려 전제로 깔고 시작한다.** SaaS가 파는 것은 발급 *이후*의 검증·공유·확산이고, 내가 풀어야 할 문제는 발급 *그 자체*의 자동화와 정확성이다. 실사용 교육 담당자들도 *"증명서 커스터마이징이 특히 제한적"* 이라고 증언한다.
> **고용24 기관 API에 직접 붙어 명단을 당겨오는 것은 RAPA만 할 수 있고, 그래서 직접 만든다.** → [`research.md`](research.md) · [`AUDIENCES.md`](AUDIENCES.md)

## 🎯 핵심 기능 (1번 목표) — 고용24 API로 발급하고, 구글시트에 자동으로 쌓는다

**이 앱이 하는 일을 한 줄로 줄이면 이것이다. 나머지 기능은 전부 이 흐름의 확장이다.**
대표 증서는 **수강확인증** — 수료증은 과정이 끝나야 나오지만, 수강확인증은 **과정 진행 중에 상시로 요청된다**(훈련생의 재직·재학 증빙, 각종 제출 서류). 가장 자주·가장 산발적으로 발급되는 증서라 수기 반복 부담이 가장 컸던 지점이고, 명단 API가 붙었을 때 가장 크게 절약되는 지점도 여기다.

### 무엇이 자동화되었나 — 이전(수기) vs 지금

| 단계 | 이전 (전부 수기) | 지금 (앱) |
|---|---|---|
| 훈련생 명단 확보 | 명단 파일을 찾아 열거나 담당자에게 요청 | **과정을 고르면 고용24 `_4` 기관전용 API가 이름·생년월일을 자동 조회** |
| 성명·생년월일 입력 | 보고 옮겨 적음 ← **오타 발생 지점** | 드롭다운에서 **고르면 채워짐. 타이핑 0** |
| 과정 정보(기간·일수·시간·사업명) | 공문·사업계획서에서 찾아 옮겨 적음 | 고용24 `_2` 상세로 **자동 채움** |
| 증명서 번호 채번 | 대장에서 마지막 번호를 눈으로 확인해 +1 ← **중복 발생 지점** | **서버가 대장을 읽어 자동 채번** (`RAPA26-AXDX-####`) |
| 증명서 작성 | 양식 파일에 직접 타이핑 | 입력값이 **공식 A4 양식에 실시간 렌더링** |
| 발급대장 기입 | 발급 후 시트에 손으로 한 줄 추가 ← **누락 발생 지점** | **발급 = 시트 기록.** 한 번의 클릭으로 1행 자동 누적 |
| 출력 | 인쇄 | 인쇄 / PDF 저장 |

**수기 7단계가 전부 사라졌다.** 사람이 내리는 판단은 이제 *"누구에게, 어떤 증서를"* 둘뿐이다.

### 실제 동작 (구현 검증 완료)

```
프런트(index.html)          요청                  백엔드(Code.gs)가 하는 일
─────────────────────────────────────────────────────────────────────────────
과정 선택         ── GET ?action=courses ─────▶ RAPA_COURSES (운영 과정 30건)
                  ── GET ?action=coursedetail ▶ 고용24 HRDPOA60_2 (일수·시간·직종·사업명)
교육생 선택       ── GET ?action=roster ──────▶ 고용24 HRDPOA60_4 (기관전용: 이름·생년월일·출결·수료상태)
번호 자동 표시    ── GET ?action=meta ────────▶ 발급대장 마지막 행 → 다음 번호 · 담당자 목록
[발급 & 시트 기록] ─ POST (JSON) ─────────────▶ LockService 락 → appendCert_() → 구글시트 1행 기록
```

**구글시트에 기록되는 것:**
`수료순서 · 증명서 번호 · 교육생 이름 · 생년월일 · 증명서 종류 · 교육과정명 · 담당부서 · 담당자`

**번호 중복이 막히는 방식(현행):** 발급은 **스크립트 락(`waitLock` 20초) 안에서만** 실행되고, 락 안에서 시트의 사용번호 전체를 읽어 **클라이언트가 보낸 번호가 이미 쓰였으면 서버가 계산한 번호로 대체**한다. 동시에 발급을 눌러도 같은 번호가 두 번 나갈 수 없다.

**API 키는 프런트에 없다.** 고용24 `authKey`는 Apps Script 스크립트 속성에만 있고, 브라우저는 웹앱 URL만 안다.

## 현재 상태 (v1 — 이미 확보한 것)

앱은 이미 완성·배포되어 실제 발급에 사용 중이다. 아래는 v2의 출발점이며, **다시 만들 필요가 없는 자산**이다.

- **고용24 훈련생 명단 자동 조회** ⭐ — 과정·회차를 고르면 기관전용 API가 훈련생 **이름·생년월일·출석수·결석수·수료상태**를 내려준다. 명단 업로드도, 옮겨 적기도 없다.
- **발급 = 구글시트 대장 기록** ⭐ — 발급 한 번에 증서 생성과 발급대장 1행 기록이 **함께** 일어난다. 대장에 손대는 일이 없다.
- **증명서 번호 자동 채번** ⭐ — `RAPA26-AXDX-####`. 동시 발급 잠금(`LockService`)과 사용번호 검증으로 중복을 막는다. 사람이 번호를 매기지 않는다.
- **증명서 5종** — **수강확인증(주력)** · 수료증 · 참여확인서 · 프로젝트 우수상 · 모범상
- **과정 정보 자동 채움** — 교육기간·훈련일수·훈련시간·사업명·담당자
- **공식 양식 미리보기 + 인쇄/PDF** — RAPA 로고·협회 직인이 들어간 A4 양식
- **API 키 서버측 보관** — 고용24 `authKey`는 Apps Script에만 있고 프런트에 노출되지 않는다

⭐ = 핵심 기능 3종. 위 「1번 목표」 흐름을 구성하는 축이며, v2에서도 **교체 대상이 아니라 확장 대상**이다.

**운영 실적 (2026-07-13 실측): 누적 720건 발급, 다음 번호 `RAPA26-AXDX-0721`, 번호 중복 0건.**
→ **수기 발급 대비 720회 분량의 옮겨적기·수기 채번·대장 기입이 실제로 사라졌다.** 핵심 기능은 이미 실전에서 증명되었다.

## v2가 필요한 이유 — 세 가지 갭

> ⚠️ **전제:** 아래 세 갭 중 어느 것도 「1번 목표」(고용24 API → 발급 → 시트 자동 기입)를 부정하지 않는다.
> 핵심 파이프라인은 720건으로 검증됐고 **그대로 간다.** v2는 이 파이프라인의 **출력물(양식)** · **처리 단위(1명→40명)** · **기록 무결성**을 보강하는 작업이다.

### 갭 1. 수료증은 발행 주체가 둘인데, 앱은 하나만 안다 (가장 큰 갭)

실제 공문 양식(`(양식) 수료증_LIG 3기.pdf`)을 뜯어보니, 수료증은 **선도기업과 RAPA의 공동 발행 문서**였다.

양식 PDF의 임베드 이미지 좌표를 실측하고 페이지를 렌더링해 확인한 실제 레이아웃:

```
제 {{수료번호}} 호

              수 료 증          ← 붓글씨풍 제목, 본문은 명조 계열

성       명 : {{성명}}
생 년 월 일 : {{생년월일}}
훈련과정명 : LIG D&A The SSEN 임베디드SW 스쿨
훈 련 직 종 : 임베디드SW엔지니어링
훈 련 기 간 : 2025.12.01.~2026.06.09.(3기, 1000H)

  위 사람은 디지털선도기업 LIG D&A와 한국전파진흥협회가 주관한
  훈련과정을 수료하였으므로 이 증서를 수여합니다.
   ↑ 이 문단 뒤에 [LIG 로고]와 [RAPA 로고]가 워터마크로 흐리게 깔린다

※「현장 실무인재 양성을 위한 직업능력개발훈련 운영규정」 제13조

              2026년 6월 9일

     주식회사                    특수법인
LIG 디펜스&에어로스페이스   한국전파진흥협회 [빨간 협회 직인 — 이미지]
     ↑ 직인 자리가 비어 있음 (인쇄 후 실물 날인)
```

**차이는 "필드 몇 개 추가" 수준이 아니라 별도 템플릿이다.** v1 수료증은 상단에 RAPA 로고 헤더가 있고, 과정 정보를 2열 표에 넣고, 고딕 서체를 쓰고, 서명란이 `한국전파진흥협회장` 한 줄이다. 공식 양식은 **헤더 로고 없음 · 표 없음 · 명조 서체 · 2단 서명란 · 본문 뒤 워터마크**다.

양식에 있는데 앱에 없는 항목: **훈련직종 · 기수/훈련시간 괄호 표기 · 법령 근거 문구 · 선도기업 정식 법인명 · 선도기업 로고 워터마크 · 2단 서명란**

선도기업은 **과정마다 다르다** (카카오 · 네오위즈 · 현대오토에버 · LIG D&A · WINS · DI&DIT …).

### 갭 2. 40명 수료하면 40번 반복해야 한다

현재는 **교육생을 한 명씩 골라 발급**한다. 40명 과정이 끝나면 같은 조작을 40번 한다. 그런데 **수료 여부(`수료상태`)는 이미 API가 알려주고 있다.** 자동 필터링과 일괄 발급을 막을 이유가 없다.

### 갭 3. 대장 기입은 자동인데, 대장 자체가 사람 손을 탄다

**자동 기입은 문제가 아니다 — 그게 핵심 기능이고 잘 돌아간다.** 문제는 기록되는 **그릇** 쪽이다.

발급대장은 **한 번 나가면 회수할 수 없는 공식 문서의 원장**이다. 그런데 구글시트는 누구나 열어서 행을 지우거나 정렬할 수 있고, 채번이 "이름이 빈 첫 행을 찾아 채우는" **슬롯 방식**이라 그 순간 어긋난다. 번호 중복 방지도 DB 제약이 아니라 **애플리케이션 락에만** 의존한다 (락은 동시 발급은 막지만, 사람이 시트를 직접 편집하는 것은 막지 못한다).

> 📌 **구글시트를 버리자는 뜻이 아니다.** 시트는 담당자가 바로 열어보고 필터·정렬로 확인할 수 있는 **운영상의 큰 장점**이 있고, 사용자에게 노출되는 대장은 계속 시트여도 된다. 갭은 *"시트를 쓴다"* 가 아니라 *"시트만 쓴다 = 무결성 보장 장치가 없다"* 이다. → 열린 질문 8번

## v2 확장 과제 (우선순위 순)

> **0순위는 "핵심 기능을 깨뜨리지 않는 것"이다.**
> 고용24 명단 자동조회 · 자동 채번 · 구글시트 자동 기입 — 이 3축은 720건으로 검증된 자산이다.
> 아래 과제는 모두 **이 3축 위에 얹는 것**이며, 어느 하나라도 이 흐름을 되돌려 수기 단계를 부활시키면 그 설계는 틀린 것이다.

### 1. 선도기업 공동 발행 양식

- 수료증·상장을 **공식 양식 템플릿으로 새로 만든다** (v1 템플릿 수정이 아니라 신규). 헤더 로고 없음 · 표 없음 · 명조 서체 · 2단 서명란.
- **과정에 매핑된 선도기업 로고를 본문 뒤 워터마크로 자동 렌더링**한다 (RAPA 로고와 나란히).
- **대표이사 직인은 앱이 찍지 않는다.** 서명란 좌측을 비워 두고, 인쇄 후 선도기업이 실물로 날인한다. **이는 현행 공식 양식이 이미 채택한 방식이다** — 양식 PDF에도 RAPA 협회 직인만 이미지로 박혀 있고 선도기업 직인 자리는 비어 있다.
- 양식 누락 필드를 채운다: **훈련직종 · 기수/훈련시간 · 법령 근거 문구 · 선도기업 정식 법인명**
- **적용 증서: 수료증 + 상장류(프로젝트 우수상 · 모범상).**
  수강확인증 · 참여확인서는 **v1 양식 그대로 유지** — 720건으로 검증된 양식이므로 건드리지 않는다.

> **핵심 발견 — 필요한 필드 대부분이 이미 API에 있다.** HRD-Net `HRDPOA60_2`를 직접 호출해 확인했다.
>
> | 양식 항목 | 출처 | 검증 결과 |
> |---|---|---|
> | 훈련직종 | `ncsNm` | LIG 3기 → **`임베디드SW엔지니어링`** (양식과 **정확히 일치**) · 카카오테크 → `SW아키텍처` · WINS → `정보보호관리·운영` |
> | 기수 | `trprDegr` | LIG → `3` (양식의 "3기") |
> | 훈련시간 | `trtm` | LIG → `1000` (양식의 "1000H") |
> | 훈련기간 | 과정 시작·종료일 | 2025.12.01. ~ 2026.06.09. 일치 |
>
> **따라서 선도기업 마스터 데이터에 사람이 넣어야 할 것은 단 2개다: 정식 법인명 + 로고 이미지.**
> (API 과정명에는 `LIG D&A` 같은 **약칭**만 있고 `주식회사 LIG디펜스&에어로스페이스` 같은 정식 법인명은 없다. 법령 근거 문구는 고정 상수.)

### 2. 수료자 일괄 발급

과정 선택 → **`수료상태` 기준으로 수료자 자동 필터** → **전원 일괄 발급** → 번호 연속 채번 → 대장 일괄 기록.
40회 조작을 **1회로** 줄인다. (`수료상태` 실제 값: `훈련중` / `80%이상수료` / `수료후취업` / `중도탈락`)

> ✅ **백엔드는 이미 절반 준비돼 있다.** `Code.gs`의 `doPost`가 `Array.isArray(body) ? body : [body]` 로 **배열 페이로드를 이미 받고**, 락 하나 안에서 `rows.map(appendCert_)` 로 여러 행을 연속 기록한다.
> 즉 **핵심 파이프라인을 뜯을 필요가 없다.** 남은 일은 (a) 프런트에서 수료자 다중 선택 UI, (b) 증서 N장 연속 렌더링/인쇄, (c) 실패 시 롤백(→ 과제 3).

### 3. 발급대장 무결성 — 시트를 지키면서 보장 장치 추가

**목표는 "구글시트를 없애는 것"이 아니라 "대장이 틀릴 수 없게 만드는 것"이다.**

- **증명서 번호 UNIQUE 보장** — 중복 불가를 애플리케이션 락이 아니라 **저장 계층이 구조적으로** 막게 한다.
- **일괄 발급 원자성** — "40장 중 37장만 기록된" 부분 실패를 없앤다.
- **슬롯 기반 채번 제거** — "이름이 빈 첫 행" 방식을 append-only 방식으로 바꾼다.

관계형 DB 이관이 정공법이지만, **시트를 사람이 보는 뷰로 남길지 / DB로 완전히 옮길지는 확정되지 않았다** → 열린 질문 8번.

## 선도기업 로고를 어떻게 보관할 것인가 — 결정과 근거

**결정: NAS를 로고 원본 보관소로 두되, 앱에는 로고를 등록해서 사용한다.**

NAS를 앱이 직접 읽는 방식은 검토 결과 **불가능**하다.

- 현재 백엔드가 **Google Apps Script**다. 코드가 **구글 서버에서 실행**되므로 RAPA 사내망 NAS로 가는 네트워크 경로 자체가 없다.
- 브라우저가 직접 읽는 것도 불가능하다. `http://` 페이지에서 UNC 경로(`\\nas\...`)나 `file://` 접근은 **브라우저 보안 정책상 차단**된다.
- NAS 웹 UI로 fetch 하는 것도 안 된다. **CORS 헤더가 없어 차단**되고 로그인 세션도 필요하다.
- Synology/QNAP 공유 링크는 대개 이미지 직링크가 아니라 **뷰어 HTML 페이지**다.

**단, 예외가 있다.** v2 백엔드를 **Node 서버 + 본인 PC 로컬 실행**으로 확정하면, 서버가 사내망 안에서 돌기 때문에 NAS를 **파일 경로로 직접 읽을 수 있다.** 이건 배포 방식과 묶인 선택이므로 아래 열린 질문에 남긴다.

**선도기업이 10개 안팎이므로 1회 등록으로 충분하다.** 등록 비용은 사실상 일회성이다.

## 사용하는 외부 API / 서비스

| # | 서비스 | 용도 | 상태 |
|---|---|---|---|
| 1 | **고용24(HRD-Net) 레거시 OpenAPI** (`hrd.work24.go.kr` 구 JSP) | ⭐ 핵심. `_1` 과정 목록 · `_2` 과정 상세(**직종·기수·시간 포함**) · **`_4` 훈련생 명단(기관전용)** | v1부터 사용, 라이브 검증 완료 |
| 2 | **Google Apps Script 웹앱** | ⭐ 핵심 백엔드. 고용24 프록시(키 노출 방지) + **발급 시 시트 1행 기록** | 배포 완료, 익명 fetch 정상 |
| 3 | **Google Sheets** | ⭐ 핵심. **발급대장 — 발급 결과가 자동 기입되는 목적지** | 운영 중, 720행 누적 |
| 4 | **관계형 DB** (Postgres 등) | 대장 무결성 보강(UNIQUE·트랜잭션) | v2 검토 중 (열린 질문 8) |

**설계 판단 — 왜 고용24 API인가 (이 앱을 가능하게 한 결정적 사실):** 공개 검색 API에는 훈련생 개인정보가 없다. 그러나 RAPA는 훈련기관이고 보유한 `authKey`가 **기관 키**이므로, 기관전용 엔드포인트 `_4`로 **자기 기관 과정의 훈련생 이름·생년월일에 접근할 수 있다.** ← **이것이 없으면 명단을 사람이 옮겨 적어야 하고, 그 순간 이 앱의 존재 이유가 사라진다.** 덤으로 `수료상태`까지 내려오기 때문에 수료 여부도 사람이 판단할 필요가 없다.

**보안:** 고용24 `authKey`는 서버측(Apps Script 스크립트 속성)에만 두고 프런트에 노출하지 않는다. `.env`·`config.js`는 커밋하지 않는다.

## 범위 밖 (v2에서 하지 않는 것)

- **선도기업 대표이사 직인 렌더링** — 앱은 로고만 찍는다. 직인은 **인쇄 후 실물 날인**으로 확정했다.
- **증명서 이메일 자동 발송** — 고용24 `_4`는 이름·생년월일·출결만 주고 **이메일·연락처를 주지 않는다.** 연락처를 별도 경로로 확보하기 전에는 불가능하다.
- **QR 코드 진위 확인 / 공개 조회 페이지** — 이번 버전에서도 **의도적으로 제외**.
- **증서 종류별 채번 분리** — v1 방식(단일 시퀀스)을 유지한다. (열린 질문)
- **다기관 지원** — `authKey`가 RAPA 기관 키이므로 명단 조회는 **RAPA 자기 과정에만** 동작한다.
- **수료생 본인 접속·자가 발급** — 담당자 전용 내부 도구다.
- **권한 관리 / 로그인**

## 열린 질문 / 리스크

1. 🔴 **수료 판정 기준 — 최우선, 개발 착수 전 확정**
   `수료상태` 값은 `훈련중` / `80%이상수료` / `수료후취업` / `중도탈락` 네 가지다. **`80%이상수료` + `수료후취업` = 수료**로 볼 것인가?
   **틀리면 엉뚱한 사람에게 공식 수료증이 나가고 회수할 수 없다.** 자동 필터는 "기본 선택값"으로만 쓰고 최종 확정은 사람이 검토하도록 설계하되, 기준 자체는 확정해야 한다.

2. 🟢 ~~직인 혼재 문제~~ → **해소.** 공식 양식 PDF를 렌더링해 확인한 결과, **현행 양식이 이미 "RAPA = 이미지 직인 / 선도기업 = 빈 자리(실물 날인)" 방식**이다. 혼재는 이미 관행이고 그 방식으로 문서가 나가고 있다. 개발 착수 블로커가 아니다.
   남는 확인: 인쇄본을 선도기업에 보내 실물 날인받는 절차가 실제로 그렇게 운영되는지 (개발과 병행 확인 가능).

3. 🟢 ~~배포 방식 ↔ NAS 연동~~ → **해소.** 로컬 실행을 택하면 NAS 접근이 가능하지만, **로고를 "NAS에서 최초 1회 임포트해 DB에 등록"** 하는 방식으로 하면 배포 방식과 로고 전략이 분리된다. 실시간 NAS 의존은 만들지 않는다.

4. 🟡 **증명서 번호를 증서 종류별로 분리할 것인가**
   현재는 5종이 번호 하나를 공유한다. 실제 공문 규칙상 종류별로 나눠야 하는지(`제2026-수료-0001호` 형태) 확인 필요. v2 범위에서는 제외했으나, **DB 이관 시점이 스키마를 바꾸기 가장 좋은 타이밍**이다. → `sequences(scope)` 테이블로 **나중에 무중단 확장 가능하게** 설계해 둔다.

5. 🟡 **v1 발급 소요 시간 미측정** — "1건 발급에 몇 초"를 잰 적이 없다. **v2로 바꾸는 순간 v1 성능은 영영 측정할 수 없다.** "40회 → 1회" 개선을 숫자로 증명하려면 착수 전에 재둬야 한다.

6. 🟡 **선도기업 로고 수급 (리드타임 최장)** — 10개사 각각 **정식 법인명 + 로고 원본**이 필요하다. 기존 공문 PDF에서 추출할 수도 있으나 **1비트 디더링 이미지라 품질이 나쁘다.** 원본(SVG/고해상도 PNG) 요청이 1순위. 타사 CI이므로 사용 승인도 함께 받는다(이미 공동 발행 문서에 쓰이는 로고라 리스크는 낮음).

7. 🟡 **과정 목록 하드코딩** — `RAPA_COURSES`가 코드에 박혀 있어 과정이 추가되면 코드를 고쳐야 한다. DB 이관 시 함께 옮길 것.

8. 🟡 **구글시트를 남길 것인가, DB로 완전히 옮길 것인가** *(핵심 기능과 직결 — 신중히)*
   **"구글시트에 자동으로 쌓인다"는 것 자체가 이 앱의 핵심 기능**이다. 시트는 담당자가 바로 열어 필터·정렬·검색할 수 있고, 앱이 죽어도 대장은 남는다 — 가볍게 버릴 자산이 아니다.
   반면 시트만으로는 UNIQUE 제약도, 일괄 발급 트랜잭션도 걸 수 없다. 선택지는 셋이다:
   - **(a) 시트 유지 + 방어 강화** — append-only 채번, 시트 보호 범위, 헤더 검증. 가장 적은 변경.
   - **(b) DB 원본 + 시트 미러** — 무결성은 DB가 보장, 시트는 읽기 전용 뷰로 계속 제공. **핵심 기능 체감이 유지되면서 무결성도 얻는다.**
   - **(c) DB 전면 이관** — 가장 깨끗하지만 담당자가 익숙한 시트를 잃는다.
   → **현재 기울기: (b).** 단, 720행 기존 대장의 이관·정합성 검증이 선행 과제다.
