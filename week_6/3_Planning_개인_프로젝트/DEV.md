## ⚠️ 착수 전 반드시 읽을 것 — 실제 양식을 뜯어보니 MISSION.md와 달랐다

DEV 계획을 세우며 `(양식) 수료증_LIG 3기.pdf`를 **이미지 단위로 분해**(임베드 이미지 7개 + 배치 좌표 + 렌더링)했다. 그 결과 **MISSION.md의 양식 서술이 실제와 다르다**는 것을 확인했다. 이 차이가 v2 렌더링 설계 전체를 좌우하므로 먼저 정리한다.

### 발견 1. 선도기업 로고는 "서명란"이 아니라 **본문 뒤 워터마크**다

MISSION.md는 이렇게 적었다:

```
주식회사 LIG디펜스&에어로스페이스   [선도기업 로고 + 대표이사 직인]   ← 실제로는 이렇지 않음
특수법인 한국전파진흥협회            [RAPA 로고 + 협회 직인]
```

**실제 양식은 다르다.** PDF 좌표(A4 595×841pt, 원점 좌하단) 실측:

| 이미지 | 정체 | 크기(pt) | 위치 | 실제 역할 |
|---|---|---|---|---|
| `Im7.png` | **LIG D&A 로고** | 211×38 | x=121, y=378 | **본문 뒤 워터마크(좌)** — 흐리게 |
| `Im5.jpg` | **RAPA 로고** | 117×89 | x=359, y=357 | **본문 뒤 워터마크(우)** — 흐리게 |
| `Im3.jpg` | **RAPA 협회 직인**(빨간 전서체) | 74×72 | x=427, y=125 | **서명란, 이미지로 인쇄됨** |
| `Im1`,`Im2` | `｢ ｣` 괄호 글리프 | 14×15 | y=282 | 법령 문구 줄 |

즉 로고 2개는 **본문 문단 뒤에 나란히 깔린 배경 워터마크**이고, 하단 서명란은 **로고 없이 텍스트 2단**이다.

```
              2026년  6월  9일

        주식회사                      특수법인
  LIG 디펜스&에어로스페이스      한국전파진흥협회 [빨간 직인 이미지]
        ↑ 직인 없음(빈 자리)              ↑ 직인이 이미 이미지로 찍혀 있음
```

### 발견 2. **직인 혼재 문제의 답이 양식 자체에 있다** — 블로커 해소

**공식 양식 PDF에 이미 RAPA 협회 직인이 이미지(`Im3.jpg`)로 박혀 있고, 선도기업 대표이사 직인 이미지는 존재하지 않는다.** 즉 **현행 공식 양식이 이미 "RAPA=이미지 직인 / 선도기업=빈 자리(실물 날인)" 혼재 방식**이고, 이 방식으로 이미 문서가 나가고 있다.

→ **앱은 선도기업 직인을 렌더링하지 않는다.** 이 리스크는 개발 착수를 막지 않는다.

### 발견 3. 수료증 양식은 v1 레이아웃과 **구조가 다르다** (필드 추가 수준이 아님)

v1이 실제 발급한 문서(`카카오부트캠프 클라우드 4기 수강확인증_김가령 님.pdf`)와 공식 수료증 양식을 렌더링해 대조했다.

| 항목 | v1 현재 (`index.html`) | 공식 수료증 양식 | v2 작업 |
|---|---|---|---|
| 상단 헤더 | RAPA 로고 + 협회명 + 영문명 | **없음** | 수료증에서 **제거** |
| 필드 | 성명 / 생년월일 (2개) | 성명 / 생년월일 / **훈련과정명 / 훈련직종 / 훈련기간** (5개) | 3개 **추가** |
| 과정 정보 | **표**(교육과정명 \| 교육기간 2열) | **표 없음** — 라벨:값 목록 | 표 **제거** |
| 훈련기간 표기 | `2026.05.12.~2026.11.17` + `(125일/1000시간)` | `2025.12.01.~2026.06.09.(3기, 1000H)` | 기수·시간이 **괄호 suffix**로 합쳐짐 |
| 본문 | "…와 한국전파진흥협회가 **공동운영하는** 아래 교육과정을 성실히 이수하였으므로…" | "위 사람은 **디지털선도기업** LIG D&A와 한국전파진흥협회가 **주관한 훈련과정을 수료**하였으므로…" | 문구 **교체** |
| 법령 근거 | 없음 | `※「현장 실무인재 양성을 위한 직업능력개발훈련 운영규정」 제13조` | **추가**(고정 상수) |
| 워터마크 | 없음 | **선도기업 로고 + RAPA 로고** (본문 뒤, 흐리게) | **신규** |
| 서명란 | 1단 — `한국전파진흥협회장` + 직인 | **2단** — 좌:`주식회사 / LIG 디펜스&에어로스페이스`(직인 없음) · 우:`특수법인 / 한국전파진흥협회` + 직인 | **재설계** |
| 발행주체 표기 | 한국전파진흥협회**장** | 특수법인 한국전파진흥협회 (**"장" 없음**) | 수료증만 **변경** |
| 서체 | 고딕 (`Malgun Gothic`) | **명조/바탕 계열**, 제목은 붓글씨풍 | 수료증·상장만 **명조로** |

**즉 수료증은 "필드 몇 개 추가"가 아니라 별도 템플릿을 새로 만드는 작업이다.**
단, **수강확인증·참여확인서는 v1 레이아웃 그대로 유지**(RAPA 단독 발행, 720건 발급 실적으로 검증된 양식). 건드리지 않는다.

### 발견 4. 서체 불일치가 "양식 100% 재현" 목표의 실질 리스크다

v1은 `Malgun Gothic`(**고딕**)을 쓴다. 공식 양식은 **명조/바탕 계열 serif**이고 "수 료 증" 제목은 붓글씨풍이다. → Windows 기본 탑재 `Batang`(바탕) / `Gungsuh`(궁서)로 매칭. **Phase 1에서 먼저 확정**한다(폰트를 나중에 바꾸면 레이아웃이 통째로 틀어진다).

⚠️ **다중 사용자가 되면서 폰트가 클라이언트 의존이 되었다.** 증서는 **담당자 PC의 브라우저**에서 렌더·인쇄된다. `Batang`/`Gungsuh`는 Windows 기본 폰트지만, **담당자 중 Mac 사용자가 있으면 폰트가 깨진다.** → Phase 0에서 담당자 PC OS를 확인하고 `font-family` 폴백 체인을 명시할 것.

### 발견 5. 선도기업 로고는 **기존 공문 PDF에서 추출 가능**하다 (단, 품질 주의)

이번에 LIG D&A 로고를 `(양식) 수료증_LIG 3기.pdf`에서 실제로 추출했다. 과정별 공식 양식 PDF를 보유 중이라면 로고 수급의 가장 빠른 경로다.

⚠️ **단, 추출본은 품질이 나쁘다.** `Im7.png`은 1633×292의 **1비트 디더링(스크린톤) 이미지**다. → **원본(SVG/고해상도 PNG)을 선도기업에서 받는 것이 1순위**, PDF 추출본은 폴백.

## Requirements

**v2 신규 — 양식**
- [ ] 수료증·프로젝트 우수상·모범상 3종이 **선도기업 공동 발행 양식**으로 렌더링된다
- [ ] 과정에 매핑된 **선도기업 로고가 자동으로 워터마크 렌더링**된다
- [ ] 양식 필드 100% 재현: 훈련직종(`ncsNm`) · 기수(`trprDegr`) · 훈련시간(`trtm`) · 법령 근거 문구 · 선도기업 정식 법인명 · 2단 서명란
- [ ] **훈련직종 수기 입력 0건** — `_2` API `ncsNm`에서 자동 채움

**v2 신규 — 일괄 발급**
- [ ] 과정 선택 → **`수료상태` 기준 수료자 자동 필터** (수동 분류 0건)
- [ ] 필터된 명단을 **사람이 검토·수정한 뒤** 확정 발급한다 (공식 문서이므로 무검토 자동 발급 금지)
- [ ] **이미 발급된 사람은 검토 화면에 표시**되고 기본 체크 해제된다 *(다중 사용자 → 중복 발급이 실제 위험)*
- [ ] 수료자 **전원 일괄 발급이 1회 조작**으로 완료된다 (현재 40회)
- [ ] 일괄 발급 시 번호가 **연속 채번**되고 대장에 **일괄 기록**된다
- [ ] **40장을 한 번에 인쇄/PDF 저장**할 수 있다 (담당자 브라우저에서 40페이지 연속 출력)

**v2 신규 — 대장 / 다중 사용자**
- [ ] 발급대장이 **관계형 DB(SQLite)** 로 이관된다
- [ ] 증명서 번호 중복 불가가 **DB UNIQUE 제약으로 구조적 보장**된다 (애플리케이션 락 의존 탈피)
- [ ] 일괄 발급이 **트랜잭션**으로 처리된다 — 실패 시 **부분 기록 0건**
- [ ] 🔴 **누가 발급했는지(`issued_by`)가 기록된다** — v1엔 없던 항목
- [ ] 🔴 **언제 발급했는지(`created_at`)가 기록된다** — v1엔 없던 항목
- [ ] 🔴 **여러 담당자가 동시에 접속·발급해도 번호가 겹치지 않는다** (WAL + `BEGIN IMMEDIATE`)
- [ ] 기존 **720건이 손실·번호 어긋남 없이** 이관되고, **다음 번호가 정확히 `RAPA26-AXDX-0721`**이다
- [ ] 🔴 **NAS에서 상시 실행**되어 담당자들이 브라우저로 접속해 쓴다

**v1 유지 (회귀 금지)**
- [ ] HRD-Net `authKey`가 **프런트에 노출되지 않는다** (서버측 보관)
- [ ] 훈련생 명단 자동 조회 (`_4`) — 명단 수기 입력 0건
- [ ] 증명서 5종 발급 유지
- [ ] 수강확인증·참여확인서는 **기존 양식 그대로** (변경 금지)
- [ ] 발급 1회 = 대장 1행 기록

## Non-goals

- **로그인 / PIN / 권한 관리** — **넣지 않는다.** 사내망 전용 + 신뢰 기반.
  단 **"누가 발급했는지"는 기록한다**(드롭다운 선택). 인증(authentication)이 아니라 **기록(attribution)** 이다. 둘을 혼동하지 말 것.
- **선도기업 대표이사 직인 렌더링** — 인쇄 후 실물 날인 (발견 2로 재확인)
- **증명서 이메일 자동 발송** — `_4` API가 이메일·연락처를 주지 않는다
- **QR 진위확인 / 공개 조회 페이지**
- **증서 종류별 채번 분리** — 단일 시퀀스 유지. 단 **스키마는 확장 가능하게 설계**(`sequences.scope`)
- **다기관 지원** — `authKey`가 RAPA 기관 키
- **수료생 본인 자가 발급**
- 🔴 **NAS에서 서버측 PDF 생성(puppeteer)** — **하지 않는다.** NAS에 Chromium은 무겁고, 클라이언트 브라우저 인쇄로 충분하며, **NAS에 한글 폰트를 깔 필요도 없어진다**

## Style

**두 개의 서로 다른 디자인 언어가 한 화면에 공존한다. 섞지 말 것.**

| 영역 | 스타일 |
|---|---|
| **조작 UI** (입력폼, 상단바, 일괄 발급 화면) | Apple 스타일 유지 — 흰 배경, `#0071e3` 블루, `rounded-xl`/`rounded-full`, `backdrop-blur`. v1 그대로 |
| **증명서 본체** (`.cert-page`) | **격식 있는 공식 문서.** Apple 스타일 **금지**. A4 794×1123px 고정 |
| 수강확인증 · 참여확인서 | v1 양식 **그대로** (고딕, 헤더 로고, 2열 표) — 720건 검증됨 |
| 수료증 · 우수상 · 모범상 | **공식 양식** — **명조/바탕 serif**, 헤더 로고 없음, 표 없음, 워터마크, 2단 서명란 |

## Key Concepts

| 용어 | 뜻 |
|---|---|
| **증명서 번호** | `RAPA26-AXDX-####`. 5종이 **하나의 시퀀스** 공유. 연도 리셋 없음. 현재 720 → 다음 **0721** |
| **수료순서** | 대장 일련번호. 증명서 번호 뒷자리와 1:1인지 **마이그레이션에서 검증** |
| **수료상태** (`trneeSttusNm`) | `_4` API 값: `훈련중` / `80%이상수료` / `수료후취업` / `중도탈락` |
| ✅ **수료 판정** | **`80%이상수료` + `수료후취업` = 수료** **(확정)**. `훈련중`·`중도탈락`은 제외 |
| 🔴 **`manager` (증서 명의)** | **증서에 인쇄되는 담당자.** HRD-Net `_2`의 `trprChap`(과정 담당자)이 자동 채워지고 드롭다운으로 변경 가능. **v1 시트의 `담당자` 컬럼이 바로 이것** |
| 🔴 **`issued_by` (실제 발급자)** | **발급 버튼을 누른 사람.** 대장에만 기록되고 **증서에는 인쇄되지 않는다.** **v1엔 아예 없던 개념** |
| | ⚠️ **둘은 다른 사람일 수 있다.** 혼자 쓸 땐 문제가 없었지만, 다중 사용자에선 "누가 이 증서를 냈나"를 `manager`로 알 수 없다 |
| **선도기업** | 과정을 RAPA와 공동 운영하는 기업. 과정마다 다름. 10개 안팎 |
| **약칭 vs 정식 법인명** | 본문엔 약칭(`LIG D&A`, API 과정명에 있음), 서명란엔 정식 법인명(`주식회사 LIG디펜스&에어로스페이스`, **수동 등록**) |
| **워터마크** | 본문 뒤에 흐리게 깔리는 로고 2개. 서명란 로고가 **아님** |
| **일괄 발급 배치** | 한 번의 조작으로 생성된 발급 묶음. `batch_id`로 추적. 실패 시 통째로 롤백 |
| **`_1` / `_2` / `_4`** | HRD-Net 레거시 JSP. `_1`=과정목록, `_2`=과정상세(**직종·기수·시간**), `_4`=훈련생 명단(**기관전용**) |

## 🔴 SQLite + NAS — 반드시 지켜야 할 한 가지

SQLite가 깨지는 가장 흔한 방식이 정확히 이 구성에서 나온다. **아래 구분을 절대 헷갈리지 말 것.**

| | 구성 | 판정 |
|---|---|---|
| ✅ **우리 방식** | **Node 앱이 NAS 위에서 실행**되고 `ledger.db`도 **같은 NAS 로컬 디스크**에 있다. 담당자는 **브라우저로 접속**만 한다 | 🟢 **안전.** DB 파일 접근이 네트워크를 안 거친다. **프로세스 하나가 DB를 단독 소유**한다 |
| ❌ **절대 금지** | 각 담당자 **PC에서 앱을 실행**하고 `ledger.db`만 **NAS 공유 폴더(SMB/NFS)** 에 두고 읽는다 | 🔴 **DB 파괴 위험.** 네트워크 파일시스템은 파일 락을 신뢰할 수 없다. SQLite 공식 문서가 경고하는 고전적 실수. **발급대장이 조용히 손상된다** |

"NAS에 DB를 둔다"는 말이 두 가지로 해석될 수 있어 못 박는다. **DB에 접근하는 프로세스는 NAS에서 도는 Node 하나뿐이어야 한다.**

## 🔴 동시 발급 — WAL만으로는 부족하다

v1은 Apps Script `LockService`로 동시 발급을 막았다. v2는 DB가 막는다. 하지만 **WAL만 켜서는 채번 경합이 안 풀린다.**

```js
// single.js — DB 초기화
db.pragma('journal_mode = WAL');    // 읽기와 쓰기가 서로를 막지 않음 (다중 사용자 필수)
db.pragma('busy_timeout = 5000');   // 쓰기 잠금 대기 5초 (없으면 즉시 SQLITE_BUSY로 실패)
db.pragma('foreign_keys = ON');

// 채번 트랜잭션은 반드시 .immediate — BEGIN IMMEDIATE
const issueBatch = db.transaction((rows, batchId, issuedBy) => { /* ... */ }).immediate;
```

**왜 `BEGIN IMMEDIATE`인가:** 기본 트랜잭션(`BEGIN DEFERRED`)은 `SELECT last_no`에서 **읽기 락만** 잡고 `INSERT` 시점에 쓰기 락으로 **승격**을 시도한다. 두 담당자가 동시에 발급하면 **양쪽 다 읽기 락을 쥔 채 승격을 못 해 교착**하고, SQLite는 한쪽을 `SQLITE_BUSY`로 죽인다(재시도해도 같은 일이 반복). `BEGIN IMMEDIATE`는 **트랜잭션 시작 시점에 바로 쓰기 락**을 잡아 두 번째 요청을 얌전히 대기시킨다.

→ **채번이 직렬화되고, 번호는 겹치지 않으며, 아무도 실패하지 않는다.** `UNIQUE(cert_no)`는 그 위의 마지막 안전망이다.

**일괄 발급 트랜잭션** — `better-sqlite3`의 `db.transaction()`은 예외 시 **자동 롤백**:

```js
const issueBatch = db.transaction((rows, batchId, issuedBy) => {
  const s = db.prepare(`SELECT prefix, pad, last_no FROM sequences WHERE scope='global'`).get();
  let n = s.last_no;
  const out = [];
  for (const r of rows) {
    n += 1;
    const certNo = s.prefix + String(n).padStart(s.pad, '0');
    insertStmt.run({ ...r, seq: n, cert_no: certNo, batch_id: batchId, issued_by: issuedBy });
    out.push({ ...r, certNo });                    // UNIQUE 위반 시 throw → 전체 롤백
  }
  db.prepare(`UPDATE sequences SET last_no=? WHERE scope='global'`).run(n);
  return out;
}).immediate;                                      // ★ BEGIN IMMEDIATE — 동시 발급 직렬화
// → 40건 중 1건이라도 실패하면 전부 롤백 = 부분 기록 0건 (MISSION 목표 충족)
```
